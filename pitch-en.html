01 · The problem

Quantity takeoff is days of specialist work — and a common source of calculation error.

  • 01

    Mid-size projects carry around 300 line items

    Virket 8 alone: 42 structural drawings, each with rebar schedules, concrete volumes, embeds and fixings to tally by hand.

  • 02

    Manual reading is slow and error-prone

    Calculators sight-read rebar diameters, concrete thickness and fastener details, then transcribe into Excel templates.

  • 03

    Revisions live in loose spreadsheets

    KPM revisions track changes across drawings; most teams keep these in ad-hoc Excel files with no link back to the drawing region.

~300
line items

Per mid-size structural project. Each one has to be extracted, verified, and exported.

Let the AI take the first pass. The calculator stays in control.

02 · The solution, in one image

From PDF drawing to Wikells Excel in minutes.

A vision-AI pipeline reads the drawing at 300 DPI, extracts rebar, concrete and details with source traceability, lets the calculator verify inline, and exports directly to Wikells.

03 · User journey

Three steps — drawing in, verified BOM out.

01 — Import

Drop the K-drawing PDF

Multi-file drop, auto-split merged PDFs, AI-assisted K-code naming. Drawing index ready in seconds.

02 — Analyze

AI vision extracts the BOM

300 DPI reading, autonomous zoom, hierarchical items with Wikells chapter codes. Every item linked to its region.

03 — Verify & export

Calculator verifies, exports to Wikells

AI rows flagged for review, inline edit / accept / reject, revision history preserved. One-click Excel export.

05 · Annotation overlay

See exactly what the AI saw.

Every BOM row pins to a region on the drawing. No black-box numbers.

Region-linked, per row

One click from a BOM row jumps to the exact rectangle the number came from — and back again.

Colour-coded by material type

Rebar, concrete, fasteners, elevations, components — each layer has its own palette. Toggle off to declutter, back on for review.

AI and manual share one layer

Calculator-drawn marks sit next to AI-proposed ones with provenance preserved; you always know which is which.

06 · AI analysis and traceability

Every line item keeps its receipts.

Every BOM item links back

To the exact drawing, page, and region it was extracted from — one click to jump to the source.

AI items flagged "needs review"

Calculator sees at a glance which rows are AI-proposed vs. confirmed. Confidence ranking per item.

Inline verify, reject, or edit

Keyboard-driven review. The human stays the calculator of record.

Revision history preserved

KPM changes are tracked across drawing revisions; nothing is lost between iterations.

07 · Cross-drawing analysis

The AI follows references to pull in the right numbers.

Concrete classes, rebar grades, and standard-detail specs rarely live on the drawing in front of you. The tool reads the referenced drawing, attributes the spec back to the BOM row.

SOURCE DRAWING

K-15-6-101 — foundation section, tie-rod detail

Shows rebar Ø16 s150 around the tie-rod, references "MATERIAL OCH ANVISNINGAR per K-01-0-001". Material grade not on this sheet.

↓  AI opens the referenced drawing
REFERENCED DRAWING (followed autonomously)

K-01-0-001 — general notes

Resolves K500 C-T loose-rebar quality and NK500AB-W mesh grade. Attributed to the source drawing's BOM rows.

Sibling K-15-6-401 flagged for the calculator, not chased.

Autonomous cross-reference following

When a drawing says "MATERIAL per K-01-0-001", the tool opens that sheet, reads the spec, and resumes the source extraction — no prompting needed.

Grades stored where they belong

Rebar quality is recorded against the general drawing and attributed back to the detail's BOM rows — never invented on the detail sheet.

Siblings flagged, not chased

Adjacent references that don't contribute to the row at hand stay flagged for the calculator — no wasted analysis budget.

08 · Configurable AI provider

Swap providers per project — from the GUI.

GUI configuration, no file editing

Model, API key, and max tokens all set from a settings modal.

OpenAI or Anthropic

GPT-5.4 vision or Claude Opus 4.7 — pick the right model per project.

Key stored locally at mode 0600

Never uploaded, never logged. Only the user's account has read access.

Swap per project or customer

Different customers can have different AI contracts — one settings change, instant switch.

09 · Architecture — local-first

Runs locally. No customer data leaves the computer.

GDPR by design

A single .exe on the calculator's laptop

FastAPI backend and React frontend packaged with pywebview. JSON on disk — no database, no sync. Drawings are sent only to the chosen AI provider for analysis.

No cloud storage of calculation data

Customer project files never leave the calculator's machine.

Transparent to the user

Every outbound request is visible; the chosen AI provider is always explicit.
